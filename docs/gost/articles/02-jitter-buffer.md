# Статья 2

## Адаптивный jitter-буфер и оптимизация качества голоса в веб-конференциях

**Автор:** Разработчик проекта Voice Conference
**Дата:** 2026-07-16

---

## Аннотация

Описываются методы обеспечения непрерывности аудиопотока в веб-конференцсвязи при нестабильном сетевом канале. Рассматриваются jitter-буфер, перекрытие аудиочанков и их влияние на задержку и качество голоса. Приведены результаты тестирования на мобильных устройствах с медленным интернетом.

**Ключевые слова:** jitter buffer, Web Audio API, аудиочанки, перекрытие чанков, задержка, качество речи

---

## 1. Введение

В реальных сетевых условиях пакеты приходят с неравными интервалами (jitter). При прямом воспроизведении это приводит к:
- Рывкам звука (gap) при запаздывании пакета
- Наложению чанков при ускорении приёма

Jitter-буфер решает эту проблему, накапливая чанки перед воспроизведением. Типичные реализации VoIP используют 20-60 мс. Однако в веб-конференцсвязи задержка уже высока (WebRTC peer-to-peer), поэтому jitter-буфер может быть агрессивнее.

## 2. Архитектура jitter-буфера

### 2.1. Фиксированный буфер

```javascript
const BUFFER_SIZE = 10; // чанков
const chunkDuration = 0.04; // 40 мс

function addToBuffer(int16Data) {
  audioBuffer.push(int16Data);
  if (audioBuffer.length > BUFFER_SIZE) {
    audioBuffer.shift(); // удаление самого старого
  }
  schedulePlayback();
}
```

**Характеристики:**
- Задержка: 10 × 40 мс = 400 мс
- Устойчивость: Высокая — выдерживает jitter до 400 мс
- Недостаток: Постоянная задержка даже при стабильном канале

### 2.2. Адаптивный буфер (эксперимент)

```javascript
// Ранее использовался, в текущей версии отключён
const minBuffer = 4;
const maxBuffer = 15;
let targetBuffer = 6;

function adaptBuffer() {
  const jitter = calculateJitter(); // std deviation arrival time
  if (jitter > 100) targetBuffer = Math.min(targetBuffer + 1, maxBuffer);
  else if (jitter < 30) targetBuffer = Math.max(targetBuffer - 1, minBuffer);
}
```

**Причина отказа:** Адаптивный алгоритм создавал резкие скачки буфера, что приводило к подвисаниям на слабых устройствах (мобильные телефоны). Фиксированный буфер стабильнее.

## 3. Перекрытие аудиочанков

### 3.1. Проблема разрывов

При последовательном воспроизведении чанков (one-by-one) между ними возникают микроинтервалы тишины (1-5 мс), которые воспринимаются как «щелчки» или «прерывания».

### 3.2. Решение — overlap 10%

Следующий чанк начинается раньше завершения текущего на 10% (4 мс):

```javascript
const overlapRatio = 0.10;
const overlapTime = chunkDuration * overlapRatio; // 0.04 * 0.10 = 0.004 сек

function schedulePlayback() {
  const source = audioCtx.createBufferSource();
  source.buffer = audioBuffer;
  source.connect(playGain);

  const startTime = nextPlayTime - overlapTime; // начинаем на 4мс раньше
  source.start(startTime);

  nextPlayTime = startTime + chunkDuration;
}
```

**Результат:** Звук непрерывный, без щелчков между чанками.

### 3.3. Сравнение методов перекрытия

| Метод | Качество | Сложность | Стабильность |
|---|---|---|---|
| Без перекрытия | Щелчки | Минимальная | Высокая |
| overlap 10% | Непрерывный | Низкая | Высокая |
| GainNode crossfade | Непрерывный | Средняя | Низкая* |

*GainNode crossfade сломал воспроизведение — AudioBufferSourceNode не поддерживает перекрытие через отдельные GainNode.

## 4. Fade-in на захвате

### 4.1. Проблема

При начале записи с неотфильтрованным микрофоном первый сэмпл может быть не равен нулю, создавая клик при преобразовании в PCM.

### 4.2. Решение

```javascript
const FADE_IN_SAMPLES = 4;

function fadeInBackground(buffer) {
  for (let i = 0; i < Math.min(FADE_IN_SAMPLES, buffer.length); i++) {
    buffer[i] *= i / FADE_IN_SAMPLES;
  }
}
```

**4 сэмпла = 0.5 мс при 8 кГц.** Достаточно для подавления клика без влияния на восприятие слова (атака согласных сохраняется).

При длительном fade-in (16 сэмплов = 2 мс) наблюдается эффект «обрезания» начала слов.

## 5. Измерение задержки

### 5.1. Компоненты задержки

| Компонент | Задержка |
|---|---|
| Захват аудио (ScriptProcessorNode) | 40 мс |
| Ресемплинг + кодирование | < 1 мс |
| WebSocket передача (LAN) | 1-5 мс |
| WebSocket передача (WAN, Россия) | 20-100 мс |
| Jitter-буфер | 400 мс (фиксированный) |
| **Итого (LAN)** | **~441-446 мс** |
| **Итого (WAN)** | **~461-546 мс** |

### 5.2. Субъективная оценка

| Задержка | Оценка |
|---|---|
| < 150 мс | Отлично |
| 150-300 мс | Хорошо |
| 300-500 мс | Приемлемо |
| 500-800 мс | Плохо |
| > 800 мс | Неприемлемо |

Наша задержка (441-546 мс) попадает в диапазон «приемлемо/плохо». Это компромисс: увеличение jitter-буфера стабилизирует воспроизведение на слабых устройствах, но увеличивает задержку.

## 6. Результаты тестирования

### 6.1. Тест: мобильный телефон (Android 10, 4G)

| Параметр | Буфер 4 | Буфер 10 |
|---|---|---|
| Разрывы звука | 12 за 5 мин | 0 за 5 мин |
| Задержка | 160 мс | 400 мс |
| Ощущение разговора | Разрывы мешают | Нормально |

### 6.2. Тест: компьютер (Chrome, Ethernet)

| Параметр | Буфер 4 | Буфер 10 |
|---|---|---|
| Разрывы звука | 0 за 5 мин | 0 за 5 мин |
| Задержка | 160 мс | 400 мс |
| Ощущение разговора | Отлично | Хорошо |

**Вывод:** Буфер 10 оптимален для мобильных устройств. На стационарных компьютерах можно было бы уменьшить, но единая конфигурация проще.

## 7. Заключение

Фиксированный jitter-буфер на 10 чанков с перекрытием 10% обеспечивает непрерывное воспроизведение на большинстве клиентов при задержке ~400 мс. Fade-in в 4 сэмпла подавляет клики без влияния на восприятие речи. Дальнейшее снижение задержки возможно при переходе на AudioWorklet и адаптивный буфер.
