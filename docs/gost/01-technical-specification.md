### 1.4 Область применения

Система предназначена для организации голосовых конференц-связи по защищённому каналу в масштабе одного сервера. Доступ к конференции осуществляется по коду (PIN), без регистрации пользователей. Система не хранит персональные данные и не ведёт логирование содержания аудиопотоков.

### 1.6 Исходные данные

Проект базируется на стандартных API браузера (WebRTC getUserMedia, WebSocket, AudioWorklet, ScriptProcessorNode) и стандартных модулях Node.js. Кроссплатформенность обеспечивается Electron (десктоп) и WebView (Android).

### 1.7 Краткое описание

Система представляет собой клиент-серверное приложение, в котором сервер на базе Node.js обеспечивает маршрутизацию аудиопотоков между участниками конференции. Аудио захватывается в нативном формате устройства, конвертируется в 8 кГц, 16 бит, знаковое (Int16) и передаётся через WebSocket в виде бинарных кадров. Сервер выполняет только ретрансляцию — обработка звука не производится.

## 2. Назначение системы

### 2.1 Назначение

Система P2P Voice Conference предназначена для:

1. Организации голосовых конференций в режиме реального времени.
2. Обеспечения аудиосвязи между участниками через веб-браузер или десктопное приложение без установки дополнительного ПО на стороне клиента.
3. Предоставления вспомогательных сервисов: текстовый чат, обмен файлами, таймер длительности конференции.
4. Контроля доступа по коду (PIN) с автоматической инвалидацией по таймеру жизни (TTL).

### 2.2 Ограничения назначения

- Система не предназначена для записи и хранения аудиоданных.
- Система не обеспечивает шифрование содержимого аудиопотоков на уровне приложения (используется TLS транспортный уровень).
- Количество одновременных участников ограничено производительностью серверного узла.

### 3.2 Описание ключевых процессов

#### 3.2.1 Процесс голосовой передачи

```
Устройство → getUserMedia → AudioWorklet (resample + PCM16) → WebSocket binary frame
    → Сервер (Node.js ws) → ретрансляция всем участникам комнаты → декодирование PCM16 → Web Audio API → Динамики
```

#### 3.2.2 Процесс создания комнаты

```
Клиент → HTTP POST /api/room/create {ttl} → Сервер → генерация UUID + PIN → ответ {code, expires}
```

#### 3.2.3 Процесс подключения к комнате

```
Клиент → WebSocket /ws?code=PIN → Сервер → проверка PIN → добавление в комнату → ответ {type: "joined"}
```

## 4. Требования к производительности

### 4.1 Параметры производительности

| Параметр | Требование | Примечание |
|---|---|---|
| Задержка передачи аудио (end-to-end latency) | ≤ 100 мс | От момента захвата до воспроизведения на стороне получателя |
| Формат аудио | 8 000 Гц, 16 бит, Int16, моно | Минимально достаточный для речевого диалога |
| Размер аудиокадра | 160 байт (20 мс при 8 кГц) | Соответствует стандартному RTP payload для G.711 |
| Пропускная способность на участника | ≤ 64 кбит/с (аудио) | Не считая чат и файлы |
| Частота сердцебиения (heartbeat) | Каждые 30 секунд | WebSocket ping/pong |

### 4.2 Масштабируемость

| Параметр | Ожидаемое значение | Максимально допустимое |
|---|---|---|
| Одновременные комнаты | 10 | 50 |
| Участников в одной комнате | 5 | 20 |
| Суммарная нагрузка (CPU) | 30 % на 4-ядерном CPU | 80 % |
| Потребление оперативной памяти | 100 МБ | 512 МБ |

## 5. Требования к надежности

### 5.1 Показатели надежности

| Показатель | Требование |
|---|---|
| Доступность сервиса | ≥ 99,5 % в месяц |
| Время восстановления после сбоя | ≤ 60 секунд (автоматический перезапуск systemd) |
| Корректность ретрансляции | Потеря не более 1 % аудиокадров при штатной нагрузке |
| Отказоустойчивость отключения клиента | Одновременное отключение до 30 % участников не приводит к разрыву конференции для оставшихся |

### 5.2 Средства обеспечения надежности

1. **systemd Watchdog** — автоматический перезапуск процесса Node.js при аварийном завершении.
2. **Heartbeat WebSocket** — обнаружение разорванных соединений за ≤ 60 секунд.
3. **Graceful shutdown** — корректное закрытие всех WebSocket-соединений при остановке сервера.
4. **Отсутствие состояния на диске** — данные хранятся исключительно в оперативной памяти, что исключает повреждение файловой системы при сбое.

### 5.3 Требования к отказоустойчивости

- При падении процесса сервера systemd восстанавливает его в течение 5 секунд.
- Участники автоматически переподключаются через 3 попытки с интервалом 2 секунды.
- Комната с TTL = 0 (бессрочная) сохраняется в памяти только на время работы процесса сервера.

## 6. Требования к защите информации

### 6.1 Требования к конфиденциальности

| № | Требование | Реализация |
|---|---|---|
| ЗИ-01 | Аудиоданные не записываются и не хранятся на сервере | Данные передаются только через оперативную память, не пишутся на диск |
| ЗИ-02 | Логирование содержания разговоров запрещено | В логах фиксируются только события подключения/отключения |
| ЗИ-03 | Персональные данные не собираются | Регистрация и авторизация не предусмотрены |
| ЗИ-04 | Доступ к конференции защищён кодом (PIN) | PIN — случайный код генерируется сервером, не задаётся пользователем |
| ЗИ-05 | Передача данных зашифрована на транспортном уровне | Self-signed SSL-сертификат, TLS 1.2+ |

### 6.2 Требования к аутентификации

- Аутентификация осуществляется исключительно по коду комнаты (PIN).
- PIN генерируется криптографически стойким способом (uuid v4, 128 бит энтропии).
- Повторное использование PIN после истечения TTL невозможно.

### 6.3 Требования к защите от неразрешённого доступа

- Сервер слушает только на指定ленных портах (8443, 8080).
- HTTP-порт 8080 выполняет только перенаправление на HTTPS, не обслуживает API.
- CORS-политика ограничивает доступ по来源у.
- Rate limiting на эндпоинтах API.

## Приложение А. Формат бинарных аудиокадров

```
Структура бинарного кадра (164 байта):
┌──────────────┬──────────────────────────────────┐
│ Заголовок    │ Полезная нагрузка               │
│ (4 байта)    │ (160 байт)                      │
├──────────────┼──────────────────────────────────┤
│ Тип (1 байт) │ PCM16 mono, 8 кГц, 20 мс       │
│ Версия (1)   │ 160 сэмплов × 2 байта = 160 Б   │
│ Резерв (2)   │                                  │
└──────────────┴──────────────────────────────────┘
```

Тип кадра: `0x01` — аудио.

## Приложение Б. Формат текстовых WebSocket-сообщений

Все текстовые сообщения — JSON. Тип определяется полем `type`.

| Тип сообщения | Направление | Описание |
|---|---|---|
| `joined` | Сервер → Клиент | Успешное подключение к комнате |
| `chat` | Оба | Текстовое сообщение чата |
| `user_joined` | Сервер → Клиент | Новый участник вошёл |
| `user_left` | Сервер → Клиент | Участник покинул комнату |
| `room_expired` | Сервер → Клиент | Комната удалена по TTL |
| `error` | Сервер → Клиент | Сообщение об ошибке |
| `ping` / `pong` | Оба | Heartbeat |
