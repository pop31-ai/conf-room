# Пояснительная записка №1

## Архитектура голосового тракта системы Voice Conference

**Версия:** 2.0
**Дата:** 2026-07-16

---

## 1. Назначение документа

Настоящая записка описывает архитектуру голосового тракта системы Voice Conference — путь аудиоданных от микрофона клиента до динамика получателя, включая все этапы обработки, кодирования и маршрутизации.

## 2. Общая архитектура

Система использует **relay-модель** (не P2P). Все аудиоданные проходят через сервер:

```
Микрофон → Захват → Ресемплинг → Кодирование → WebSocket → Сервер → WebSocket → Декодирование → Воспроизведение → Динамик
```

Сервер **не обрабатывает** аудио — выполняет только маршрутизацию (ретрансляцию) бинарных фреймов всем участникам комнаты кроме отправителя.

## 3. Этапы обработки

### 3.1. Захват аудио

| Параметр | Значение |
|---|---|
| API | `getUserMedia({ audio: true })` |
| Обработчик | `ScriptProcessorNode.onaudioprocess` |
| Размер буфера | 256–8192 сэмплов (степень двойки, расчёт по 40мс от частоты AudioContext) |
| Частота дискретизации | Нативная частота устройства (обычно 44100 или 48000 Гц) |

### 3.2. Обработка перед отправкой

| Этап | Описание |
|---|---|
| **Silence gate** | Если пиковая амплитуда буфера < 0.01 — буфер отбрасывается (не отправляется) |
| **Fade-in** | Первые 4 сэмпла плавно нарастают от 0 до 1.0 (антиклип на начале фразы) |
| **Ресемплинг** | Линейная интерполяция с нативной частоты до 8000 Гц |
| **Кодирование** | Float32 → Int16: `Math.max(-32768, Math.min(32767, val * 32767))` |

### 3.3. Передача

| Параметр | Значение |
|---|---|
| Протокол | WebSocket (wss://) |
| Тип фрейма | Binary (ArrayBuffer) |
| Формат данных | Int16Array, моно |
| Частота | 8000 Гц |
| Размер чанка | ~320 сэмплов (40мс при 8 кГц) = 640 байт |

### 3.4. Маршрутизация на сервере

```
Сервер получает binary frame →
  Для каждого пира в комнате (кроме отправителя):
    if (peer.id !== senderId && peer.readyState === OPEN && peer.bufferedAmount < 2MB):
      peer.send(data)
    elif (bufferedAmount >= 2MB):
      peer.close()  // backpressure: отключение медленного клиента
```

**Приоритет:** Бинарные (аудио) отправляются немедленно. JSON (чат, файлы) — после бинарных.

### 3.5. Воспроизведение

| Этап | Описание |
|---|---|
| **Декодирование** | Int16 → Float32: `int16[i] / 32768` |
| **Создание буфера** | `AudioContext.createBuffer(1, length, 8000)` |
| **Jitter-буфер** | Фиксированный, 10 чанков максимум |
| **Планирование** | `AudioBufferSourceNode.start(nextPlayTime)` с перекрытием чанков на 10% |
| **Audio Ducking** | При активном микрофоне — gain воспроизведения снижается до 60% |

### 3.6. Audio Ducking

При захвате аудио измеряется пиковая амплитуда. Если микрофон активен (peak ≥ 0.01):

```
playGain.gain.setTargetAtTime(0.6, currentTime, 0.1)  // приглушение до 60%
```

При прекращении речи:

```
playGain.gain.setTargetAtTime(1.0, currentTime, 0.1)  // полная громкость
```

**Цель:** Подавление акустического эха (feedback loop) между динамиком и микрофоном в одном помещении.

## 4. Цепочка соединений Web Audio API

```
MicStream → MediaStreamSource → ScriptProcessorNode → Sink(gain=0) → destination
                                                                       ↓
RemoteAudio → BufferSource → playGain → destination (динамик)
                     ↓
              playGain (ducking: 0.6 при речи, 1.0 при тишине)
```

**Важно:** ScriptProcessorNode подключён к destination через silent GainNode (gain=0). Это необходимо для поддержания активности ноды — в Web Audio API ScriptProcessorNode прекращает вызов `onaudioprocess` если output не подключён к consumption-пути.

## 5. Параметры качества

| Параметр | Значение | Обоснование |
|---|---|---|
| Частота дискретизации | 8 кГц | Стандарт телефонии (G.711), покрывает речевой диапазон 300-3400 Гц |
| Квантование | 16 бит (Int16) | Динамический диапазон 96 дБ, достаточный для речи |
| Размер чанка | 40 мс | Компромисс между задержкой и накладными расходами |
| Fade-in | 4 сэмпла (0.5 мс) | Достаточно для подавления клика при сохранении атаки слова |
| Silence gate | 0.01 | Чувствительный для слабых микрофонов (бабушкины телефоны) |
| Jitter-буфер | 10 чанков (400 мс) | Запас для нестабильных каналов |
| Перекрытие чанков | 10% | Предотвращение разрывов между чанками |

## 6. Обратная связь и эхо

### 6.1. Проблема

При работе в одном помещении микрофон может улавливать звук динамиков — возникает feedback loop (акустическое эхо).

### 6.2. Решения

| Уровень | Мера | Описание |
|---|---|---|
| Серверный | Исключение отправителя | `broadcastBinary()` не отправляет аудио обратно отправителю |
| Клиентский | Audio Ducking | Приглушение воспроизведения до 60% во время речи |
| Клиентский | Отключение AEC | `echoCancellation: false` — в некоторых браузерах AEC создаёт артефакты |
| Физический | Наушники | Рекомендация: использовать наушники для полного устранения эха |

## 7. Ограничения

1. **ScriptProcessorNode** — deprecated API, блокирует main thread. Альтернатива (AudioWorklet) не совместима со всеми платформами.
2. **Отсутствие кодека** — PCM 8 кГц / 16 бит = 128 кбит/с на участника. Opus сократил бы до ~16 кбит/с.
3. **Нет AGC на сервере** — уровень громкости зависит от микрофона клиента.
4. **Фиксированный jitter-буфер** — не адаптируется к условиям сети (ранее был адаптивный, стабильнее фиксированный на слабых устройствах).
